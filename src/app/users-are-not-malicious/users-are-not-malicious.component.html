I can trust people to keep my security interests in mind

A classic mistake that programmers often make is to assume that people will only use their systems as the programmers expect them to. This assumption allows programmers to ignore issues that could arise from a system being used in an unexpected way. Unfortunately, this is a dangerous assumption. In reality, not all users are benevolent or act in good faith; some of them are actively malicious, looking for ways to exploit systems for their own benefit. Some users are unauthorized, perhaps having gained access to the system through another vulnerability, but sometimes authorized users have bad intentions (this is also known as an “insider attack”). Additionally, sometimes users are acting in good faith, but do things that the programmer did not anticipate, resulting in unexpected outcomes that can leak or corrupt data, damage hardware, deny service, or have other negative effects.

The solution to this problem is well-known, and takes the form of input validation and input sanitization. Input validation is making sure that input matches a required form or type. For example, ensuring that input meant to be a number is composed only of digits and not any other symbols. Input sanitization takes valid input and ensures that the input cannot do anything malicious or undesired. For example, input sanitization for an online “comment section” is responsible for making sure that HTML entered into a comment is shown as HTML in the comment, and not interpreted as HTML, which would modify the web page.

Code Injection

Code Injection is a broad category of attacks that involve an attacker entering executable code in an input field where code should never exist.

In an old-school attack known as SQL Injection, users purposefully paste a block of SQL code into an input field where a developer would never expect code to exist. The result might leak some secret data from an SQL database, illegally insert data, or delete data. This XKCD comic explores how user input can destroy a database. For further reading here is an explanation of the XKCD comic.

Another classic example of code injection is a Cross-Site Scripting (XSS) attack. An XSS attack could permanently embed executable code within a server, which would execute whenever someone requests a page from the server, or it could be a piece of code within a request that "bounces" off the server. The former is called a persistent attack, and the latter is called a reflected attack.

Back in the days of MySpace, a 20-year-old named Samy Kamkar launched a successful cross-site scripting attack known as the "Samy Worm," which created a pop-up saying "but most of all, Samy is my hero" and automatically sent Samy a friend request. While the vulnerability lay within web browsers, MySpace took most of the heat and eventually found a way to patch the bug themselves. Samy released an explanation of the exploit, which is archived here.

Buffer Overflow

In a buffer overflow attack, a user exploits the limited size of memory given to a function. When the code within a function finishes running, it tells the program where the next piece of code is using a return address. If a user enters more input than the function can hold, the program will start writing over other code that it needs, including the return address. If a user places a block of executable code in an input field that is longer than the input field should hold, they could write over the return address. If they rewrite the return address to point to the beginning of their executable code input, the computer will execute everything the user wrote! Read more about buffer overflows here, or check out Computerphile's video on buffer overflows if you want to see a live demonstration.

Input Validation

In 2007, researchers found a bug in a C++ library that could cause a server to crash. A normal HTTP request to a server will specify what version of HTTP the server should use to respond to the client. The C++ library checked that the HTTP version number was valid, but it did not check that the version number existed. If a client sent a request without an HTTP version number, the server would crash! Input validation should not only check that the input is of the correct type. It should also check that input exists. You can read about this vulnerability on the National Vulnerability Database.

Common Vulnerabilities and Exposures CVE

Users sometimes act very differently from how you would expect. Many will make careless mistakes when using your software, or misunderstand how to use it. Some will be malicious insiders acting in bad faith. Some “users” will actually be attackers using stolen accounts, looking for ways to compromise your system. Malicious or not, you should always assume user input is dangerous in its "raw" form. The solution to this universal problem is to validate input — check that it is the correct type and is of the expected size — and to sanitize input — make sure that valid input cannot harm the system. The good news is that many common software libraries contain robust and time-tested tools to validate and sanitize input, so you only need to know how to use them and how to recognize where they should be applied. Remember: user input is malicious until proven innocent!
